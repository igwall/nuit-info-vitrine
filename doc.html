Notre système

Le système se divise en trois sous partie, la partie mission, la partie camp de base et la partie centre de contrôle.

L’explorateur doit pouvoir contrôler ses différents outils, ces derniers seront pour la plupart des objets connectés qui transmettent d’eux-même des informations sous forme de logs. Ces équipements sont des éléments très importants pour l’explorateur, il est donc vital de les vérifier avant chaque mission où ils seront utilisés.

La partie mission permet à l’explorateur de planifier ses sorties, cette interface lui permettra d’avoir une checklist pour vérifier qu’il n’oublie rien avant de partir en mission ainsi que des protocoles qu’il devra suivre au cours de la mission. Il pourra aussi noter ses avis ou ses ressentiment durant la mission.

Lorsque tous les éléments de la checklist de la mission seront checkés, cette dernière sera complété et ne sera plus affiché dans les missions à effectuer.

L’explorateur doit aussi pouvoir prévoir des tâches ponctuelles ou répétitives dans son camp de base, pour cela il aura une interface camp de base où il pourra définir des tâches à effectuer, définir le temps de répétition de cette tâche. Ces tâches seront aussi matérialisé sous la forme d’une checklist.

Une interface centre de contrôle permet de connaître les informations sur l’explorateur ainsi que sur son environnement. Les données sur l’environnement seront récupéré via des API ou directement envoyés par des systèmes embarqués équipés de capteurs.

Comportement simulé

Afin de reproduire le comportement des outils de l’explorateur qui envoient des logs, nous avons produit des clients en python qui font des requêtes posts vers le serveur. Ces derniers sont représentés par EquipmentLog. Chaque log appartient est d’un certain type, ce type permet de traiter ces logs par importance et d’agir en conséquence.

Les checklists enregistrés sont créées par rapport à des templates de checklist. Afin de modéliser ces templates nous avons utilisé une variante de l’abstract occurrence pattern, de ce fait plusieurs checklists peuvent avoir le même modèle tout en étant traités indépendamment.

Technologies employées

Afin de réaliser cette application, ils nous a paru évident d’utiliser une architecture SOFEA. Nous avons donc séparé le côté client et serveur. Toutes les communications avec le serveur se font à la manière d’un web service à l’aide de REST. Nous avons décidé de relever le défi d’effectuer une Progressive Web App et de dockerisation afin d’accroître la flexibilité et la portabilité d’exécution de l’application.

Le Front End de l’Application est en Vue.js car c’est un framework js très performant et léger. Nous utilisons aussi Bootstrap afin d’avoir une application responsive.

Du côté Back End nous utilisons le framework Ruby on Rails car il permet de développer rapidement des applications web tout en respectant les bonnes pratiques d’usage. Notre Back End étant seulement une API faisant des appels à une base de données, la mise en place des fonctionnalités de base fut très rapide.

Déploiement

L’application est déployée sur Heroku car il propose une solution d’hébergement gratuite et facile à mettre en place. Il propose aussi d’ajouter des add-ons au projet, ce qui nous a permis d’héberger notre base de données PostgreSQL sur Heroku. Heroku propose aussi des commandes qui permettent par exemple de récupérer les logs de l’application qui est déployée ce qui est très pratique pour débugger la version déployée.

A partir du moment où l’application est déployée, il est très simple de la mettre à jour, en effet le serveur Heroku est lié au dépôt Github sur lequel on envoie les sources de l’application et il est donc possible de déployer de nouvelle version de l’application à partir d’une branche du Github lié. Il est également possible de faire en sorte que Heroku déploie automatiquement dès que l’on push sur la branche master du Github.
